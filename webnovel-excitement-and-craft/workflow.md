# 爽点与文笔 Agent

双功能合并在一个 skill：**爽点（结构层）** + **文笔（文字层）**。任何一次改写任务先诊断是哪一层的问题，再进对应分支。

## 分支决策

```
if 问题是"打脸不够爽 / 装逼没劲 / 读者不爽" → 走 PART 1 爽点
elif 问题是"描写空 / 没画面 / 文笔差 / 重复词多" → 走 PART 2 文笔
else 同时走两路，但先爽点后文笔（结构问题优先）
```

# PART 1 · 爽点生成 / 加挂

## 输入契约

```yaml
context: <这段剧情之前发生了什么，或贴原文>
conflict: <该剧情承担的矛盾；若用户没说，必须先反问>
first_excitement_of_book: <书名里暗示的第一个爽点，来自 story-blueprint>
used_excitement_types: <本作品之前已经用过哪些爽点类型；用于避免同类堆叠>
```

缺 `conflict` 一律打回。

## 爽点类型表（用于分类与查重）

```
生理向：
  武力爽：我弱小→获得惊天武技/秘籍/身份→碾压
  桃花爽：命犯桃花，异性吸引
心理向：
  虚荣爽：有钱有势有女人
  信息差爽：你不知我知（穿越者背古诗、重生者知股市）
  反击爽：扮猪吃虎 / 弱势反击
  瞩目爽：全球焦点 / 武林至尊
  嘚瑟爽：故意气人，读者隔屏笑
  威慑爽：惊天实力 + 不外露 / 不外露但又让人怕
  身份爽：别人仰视的身份被主角一眼拆穿 / 主角本身就是
```

用英文代号入库以便查重：`PWR / PEACH / VAN / INFO / REV / STAR / SMUG / AWE / ID`。

## 生成工作流

```
STEP 1  从 conflict 里提取"主角靠什么碾压对手"的能力。
STEP 2  对应到爽点类型表，候选 2 个类型（不能和 used_excitement_types 最近 2 次重复）。
STEP 3  设计爽点三段：
        (a) 压力段：让对手先得意 / 让读者替主角憋屈
        (b) 碾压段：主角用能力解决
        (c) 反应段：在场 NPC 的惊/怕/羡/妒/服 具体到动作+表情
STEP 4  把爽点挂回主线：爽点尾必须带一句"这一 X 打碎的不只是他，是……"
        → 让爽点服务更大的矛盾链。
```

## 输出 schema

```markdown
## 爽点方案 · [一句话命名]

- 挂靠矛盾：<复述 conflict>
- 爽点类型：<代号 + 中文说明>
- 为什么选这个类型：<排除同类堆叠的说明>

### 三段结构
- 压力段：<指定用户在哪段前插入，或直接写出 150–300 字>
- 碾压段：<主角的具体动作 + 能力来源；不允许"他轻描淡写一挥手"这类空写>
- 反应段：<在场 NPC 的具体反应，至少 2 个名字对应 2 种表情/动作>

### 挂回主线
- 尾句：<一句把爽点上升到主线层的收束>
```

## 爽点拒绝清单

以下任一命中 → 重写，不得交付：

- [ ] 爽点前没有压力段（直接爽）
- [ ] 反应段只写"众人惊呆"而无具体动作
- [ ] 爽点的能力来源在前文无铺垫
- [ ] 同类爽点与最近两次重复
- [ ] 爽点脱离矛盾链 → 成为挂件

# PART 2 · 文笔诊断与改写

## 四层模型

| 层 | 症状 | 修复手段 |
|---|---|---|
| L1 规范 | 错字、病句、啰嗦 | 短句化 + 拗口朗读过滤 |
| L2 准确 | 词不达意 / 复读词 | 同义替换库 + 描写五类模板 |
| L3 感染力 | 读者进不去场景 / 不共鸣 | VR 画面感三步 + 共鸣句库 |
| L4 反 AI 味 | 节拍器 / 句式复用 / 主语熵 / 情感标签 / 反应套餐 / 清单修为 / 设定说明书 / 定义体 / 算法思维链 / 爽点链条过完整 / 质量曲线过稳定 / **角色灵魂缺位** | `../references/anti-ai-tells.md` A–O 主条款 + B/E/G/H 扩展子条款 强制规避 |

诊断顺序必须 L1 → L2 → L3 → L4，不允许跳级。L4 的权重最高——即使 L1–L3 全 PASS，只要 L4 FAIL（尤其 M / N / O 元条款），另一台模型也能抓出 AI 生成痕迹，必须重写。O（角色灵魂缺位）在文笔层的表现形式是：配角 / 反派 / 动物 / 路人台词与动作 100% 围绕主角或剧情功能展开，找不到属于角色自己的内部状态渗出。

## 输入契约

```yaml
material: <用户贴的原文>
focus: regulation | precision | impact | all
keep_style_hint: <用户希望保留的风格，如"主角口语化""第一人称幽默"；可空>
```

## 诊断工作流

```
STEP 1  L1 规范
  - 标出错字
  - 标出病句（以"读出来拗口"为启发式）
  - 标出超过 30 字的长句
STEP 2  L2 准确
  - 找出重复超过 3 次的词（说/看/走/然后）
  - 找出模糊形容词（"很厉害""非常美""相当不错"）
STEP 3  L3 感染力
  - 查画面感：是否只有一个感官？能否加视/听/嗅/触中另一种？
  - 查共鸣：是否抓到读者能共情的生活经验？
STEP 4  L4 反 AI 味（必跑）
  - 对文本逐项判定 anti-ai-tells.md 的 A–O 主条款 + B/E/G/H 扩展子条款，给出 PASS/WARN/FAIL
  - 任一 FAIL 必须改，不允许只改 L1–L3 就交付
  - O 条款的诊断手法：把每个有名出场角色（含动物 / 灵兽 / 反派 / 路人）的所有台词与动作抽出来，问自己"这些行为换成另一个同阵营同身份的角色来做是否毫无违和"、"有没有任一条与当前剧情功能无关的细节"。两个问题都是"是" / 都是"否" → FAIL
STEP 5  给出对照改写（含 L4 改动理由标注）
```

## 词汇替换快查

- 说 → 道 / 讲 / 回 / 嘟囔 / 低声
- 喜欢 → 欢喜 / 喜爱 / 钟意 / 偏爱
- 尊敬 → 崇敬 / 敬爱 / 推崇
- 红 → 珊瑚红 / 砖红 / 石榴红 / 酡红 / 玫红 / 橘红 / 枣红 / 霜叶红
- 绿 → 葱绿 / 薄荷绿 / 湖绿 / 豆绿 / 柳绿 / 松花绿 / 翡翠绿 / 碧绿
- "很…" → 用具体动作替换。例："很生气" → "拳头攥出了青筋"

## 描写五类模板库

每类背后都有 5 条以上可用脚本（不是固定句式，而是"要素清单"）：

```
打扮：材质 + 颜色 + 剪裁 + 装饰 + 违和感 / 一致感
人物：体型 + 面部 + 眼神 + 标志动作 + 与环境的不协调
景色：远景 + 中景 + 近景 + 至少一个反差物
动作：预备 + 发力点 + 幅度 + 余势 + 对方反应
声音：音色 + 音量 + 节奏 + 与场景的对比
```

## VR 画面感三步（L3 核心修复手段）

```
原点 → 视觉（1 个具体细节）→ 嗅/听/触（再叠 1 层）→ 生理/情绪反应
```

**参考样本**：
> 站在弄堂口，四周漆黑一片、放眼看向深深的弄堂，幽深而死寂，恰在此时一阵冷风飘来，满鼻腔都是一股腐朽的味道，感受着后脖颈的凉意，顿时汗毛倒竖了起来。

分解：原点（弄堂口）→ 视觉（漆黑、幽深）→ 嗅觉（腐朽味）→ 触觉（凉意）→ 生理（汗毛倒竖）。

## 共鸣修复手段

当场景情绪需要读者共情，插入一句"日常共鸣句"。示例模式：

```
"<都市成年人场景>" + "<一个令人破防的日常小动作>"
```

参考样本：
> 现代社会男人承受着工作和家庭的双重压力，找不到属于自己的时间和空间，哪怕只是车开到楼下，坐在车里慢慢抽完一根烟再上楼的时间都变的如此的宝贵。
